## 1. Chaine E2E complete — ORDRE verifie

Flux trace de la source de fond jusqu'a la pastille :

    DesktopUiaPollService (IHostedService, Timer .NET 1,5 s, thread POOL)   [ROB-07]
      -> DesktopUiaSessionSource.Poll(now)
           -> IUiaTreeProvider.TryGetTree() -> MapTree(pur) -> _cache (volatile)

    SessionsViewModel.Refresh (DispatcherTimer 2 s, thread UI)
      -> SessionMonitor.Read(now)
           1)   transcripts (base, app bureau incluse)
           2)   fichiers d'etat hooks (surcharge)
           2.b) _desktop.Read(now)  -- rend le cache, non bloquant --       [ROB-07]
           2.c) foreground = _foreground?.IsClaudeForeground()  (Win32, us)
                _tracker.Observe(raw, foreground, now)   <- OBSERVE sur snapshots BRUTS
           3)   filtre archived (permanent) PUIS filtre treated (reversible)
      -> OrderBy(rank) -> Items (KindLabel + etat + couleur) -> XAML pastille

Ordre observe-AVANT-filtre : CORRECT et load-bearing. SessionMonitor.Read lignes 100-103
(Observe) s'executent AVANT les lignes 110-118 (double filtre). Indispensable : le tracker
doit voir la session en attente MEME quand elle est deja treated pour pouvoir la PURGER
(NET-03) sur un nouvel episode. Si le filtre passait d'abord, la session traitee disparaitrait
de raw, le tracker ne verrait jamais son nouvel episode d'attente et la reapparition NET-03
serait cassee. L'ordre actuel est le bon.

Non-bloquant sur toute la chaine : Read ne fait AUCUNE I/O UIA (rend _cache) ; le walk UIA
couteux vit exclusivement dans le poll de fond (thread pool) ; IsClaudeForeground() est un
couple GetForegroundWindow/GetWindowText (quelques us) ; chaque etape ajoutee (2.b, 2.c) est
sous try/catch — une source/observation defaillante ne casse jamais le pipeline CLI ni ne fige
l'overlay.
